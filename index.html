Flat JS A simpler JS framework

Or, a client side JS framework for people that don't like client side JS frameworks.

Why

Modern day JS requires application structure, but I'm not sold on MV* making the most sense for a language built to operate a user interface.

But I still needed something that aided in structuring my JS apps, so I created FlatJS.

Alternatives

Popular JS frameworks impose ideologies which may be appropriate for some applications, but not all.

I needed something that would make writing JS apps quicker, and allow me to structure my application based on the app's needs, not dogma.

Who

FlatJS is intended for anyone who writes client side JS applications.

Written and maintained by Sankho Mallik, a web developer living and working in the greater NYC area.

Any and all feedback as well as code contributions are welcome!

Easy, DOM contextual code execution.

Run JS directed by data attributes on your markup.

This allows FlatJS to be equally useful if your markup is generated on the server side or if you do client side templating.

It also forces you to organize your code with namespaces, but allows you to define the application structure's abstractions however you'd like.

FlatJS.ModuleRunner

<div data-js-module="NS.example-namespace.example-module">
  ...
  ...
  ...
</div>

Markup.

Maybe the server generates it; maybe you did with JS. Who knows. When you want to run Javascript, just add a reference in the data-js-module attribute.

FlatJS.ModuleRunner

<script>
  var NS = {

    exampleNamespace: {

      exampleModule: function(node) {
        node.innerHTML = 'Click Me';

        node.onClick = function() {
          node.style.color = '#333';
        }
      }
    }
  }
</script>

Application Code.

Structure JS however you want. The only requirement is that you use a namespace.

The ModuleRunner by default calls JS functions with the new operator. This can be changed however you want in the options.

FlatJS.ModuleRunner

<script src="/js/FlatJS.ModuleRunner.js"></script>

<script>
  var NS.runner = new FlatJS.ModuleRunner({
    // see options
  });

  NS.runner.init(document);
</script>

Load the script, run your JS.

Save the runner to your namespace so you can execute it after ajax calls. Pass any node you want instead of document to target only what you need.

Read more about the FlatJS.ModuleRunner module here.

OOP behavoir for client side JS

Private member methods + class inheritance. Extending John Resig's work.

FlatJS.Classy

What's included in FlatJS?

FlatJS is composed of three base modules to aid in the following:

  1. Programmatic execution of JS on DOM elements agnostic of how the DOM elements are generated. - FlatJS.ModuleRunner
  2. Giving JS basic OOP like behavoir, including private member functions and inheritance. - FlatJS.Classy
  3. Providing an app lifecycle to JS code based on DOM interactions. - FlatJS.Widget req. FlatJS.Classy

And several optional modules to aid in certain cases:

  1. Inter module pubsub messaging - FlatJS.Dispatch req. FlatJS.Widget extends FlatJS.Widget.prototype
  2. Window / browser monitoring, resize, scroll, more - FlatJS.WindowMonitor req. FlatJS.Widget
  3. More as I make em.

But really, why?

A: Because the current crop of JS frameworks aren't simple enough.

I don't want an abstraction layer that forces me to write more code to keep up with the abstractions it's imposing on me. Nor do I believe any high level ideologies apply easily to client side JS - mainly MV*.

But what I DO want is something which saves me time by taking away trivial, but common JS tasks. Having a reliable way to execute JS code on markup, being able to program with inheritance in mind, and having an app lifecycle for JS that deals with markup - these are things I'll almost definitely need on any client side JS project.

JS is a simple language and part of what makes it great is the developer's freedom in being able to structure their application's abstractions according to what the application needs. FlatJS tries to provide a framework that never gets in the way.

But this is all just my opinion, and in the end is just some code I've used for projects and will continue to do so personally. This website serves as a home for the project.
